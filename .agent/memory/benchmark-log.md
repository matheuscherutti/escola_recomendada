## 2026-06-28 — Benchmark Run #8 (single-source deep dive, user-requested)

**Source checked:** `garrytan/gstack` (github.com/garrytan/gstack.git). MIT licensed, owned by Garry Tan (YC President & CEO) — 118,000 stars, 17,500 forks, 297 open issues, last commit 2026-06-25 (3 days before this run), no tagged releases (rolling `VERSION` file, currently v1.58.5.0). By far the highest-credibility/highest-traction single source benchmarked to date (GitHub-verified stars, not a self-reported claim like ECC's). Same architecture class as DevBureau — markdown `SKILL.md` files synced to multiple coding-agent hosts — directly comparable. Pitched as "Garry Tan's exact Claude Code setup: opinionated tools that serve as CEO, Designer, Eng Manager, Release Manager, Doc Engineer, and QA."

**DevBureau state at time of run (per `doctor.py`):** 22 agents, 64 skills, 21 workflows.

**Structure observed:** 70 top-level dirs, 54 with their own `SKILL.md` (the rest are shared `lib/`/`scripts/`/`bin/` infra). Every `SKILL.md` is **auto-generated from `SKILL.md.tmpl`** via a `bun run gen:skill-docs` build step — gstack hand-maintains a template + frontmatter, not 54 independent prose files. 9 registered coding-agent hosts (`claude`, `codex`, `cursor`, `opencode`, `factory`, `kiro`, `slate`, `openclaw`, `hermes`) plus a 10th non-agent "host" config (`gbrain`, for memory-integration doc generation). Ships a real automated eval/CI harness for its own skills (`bun test`, `skill-llm-eval.test.ts`, `skill-e2e-*.test.ts`, `skill-routing-e2e.test.ts`, plus `eval:compare`/`eval:summary`/`eval:watch`/`eval:select` scripts) — this is the closest any benchmarked source has come to actually solving the "zero behavioral testing for skills" gap DevBureau flagged as a heavy lift back in run #5. Architecturally, gstack skills are stateful mini-programs (each ships a bash "Preamble" block doing self-update checks, session tracking, host-capability detection, telemetry-consent gating) backed by a shared runtime library — a fundamentally different shape from DevBureau's static, no-runtime-logic prose skills.

### Adopt (ready for follow-up)
1. `/gstack-upgrade`-equivalent: a self-update-aware skill/check for DevBureau's own npm/git distribution (version-drift detection, changelog-since-last-version display, backup-and-restore on failed upgrade).
2. Automated `VERSION`/`CHANGELOG.md` bump folded into `finishing-a-branch`/`/finish-branch`, replacing the current hand-written commit-message bump.
3. Per-question "stop asking me that" suppression, persisted in the memory system, addressing real Socratic Gate fatigue for returning users — scoped as a memory-system feature, not gstack's full psychographic-profile machinery.

### Consider (needs a decision)
1. `cso`-style two-cadence (daily zero-noise / monthly deep-scan), confidence-gated audit pattern for `vulnerability-scanner`, including a formalized "skill supply chain scanning" step — timely given today's SkillSpector false-positive findings, but needs a decision on cadence/scheduling since DevBureau has no built-in scheduler.
2. Multi-persona plan-review pass (CEO/Designer/EngManager/DevEx) before execution, distinct from `project-planner`'s single-voice planning — needs a decision on whether this is a new workflow step or redundant with the Socratic Gate's front-loaded questions.
3. Skill-scoped, session-invokable safety hooks (`freeze`/`careful`/`guard`-equivalent) — needs a decision on whether DevBureau's hook system should support per-skill, opt-in registration alongside its current global-only hooks.
4. `context_queries`-style frontmatter convention (filtered/sorted/limited pulls from project-scoped history files, auto-injected before a skill runs) as a lighter-weight, portable alternative to adopting `gbrain` itself — needs a decision on scope before it's actionable.
5. Periodic structured retrospective (`retro`-equivalent: mine git log + memory files into a dated, trend-tracked summary) — needs a decision on what triggers it, since DevBureau has no scheduler of its own.
6. Generalized `skillify` (codify a successful ad-hoc session flow into a reusable artifact, with explicit confirmation before saving) — needs a decision on whether this fits DevBureau's non-programmer-leaning audience or should stay power-user-only.

### Skip
1. Persistent headless-browser daemon + browser-automation subsystem (`browse`, `extension`, `pair-agent`, `connect-chrome`, `setup-browser-cookies`) — requires a long-lived local service, conflicts with DevBureau's static-markdown architecture, same reasoning that sank Headroom in run #3.

3. iOS-native on-device suite — out of scope; DevBureau's mobile coverage is RN/Flutter-first by design.
4. `SKILL.md.tmpl` + codegen build step — solves a preamble-duplication problem DevBureau's prose-only skills don't have.
5. `landing-report` (multi-workspace ship-queue dashboard) — solves a concurrent-contributor problem that doesn't exist at DevBureau's single-maintainer scale.
6. `CONTRIBUTING.md` — premature for a private repo, same precedent as runs #6/#7.
7. 23-persona "virtual team" / 54-skill catalog scale generally — lean-roster philosophy holds, same precedent as runs #1/#6/#7.

## 2026-06-26 — Benchmark Run #3 (token-economy deep dive, user-requested)

**Sources checked:** `DietrichGebert/ponytail` (markdown skills/rules, 16 agent hosts) and `headroomlabs-ai/headroom` (Rust+Python runtime context-compression proxy). User asked specifically: what can make DevBureau more token-efficient without losing code quality.

**DevBureau state at time of run (per `doctor.py`):** 22 agents, 57 skills, 16 workflows.

**Key structural finding:** the two sources operate at different layers, which changes what's portable.
- Ponytail is the same architecture class as DevBureau — plain markdown skills/rules synced into the same kind of IDE hosts (Claude, Cursor, Codex, Copilot, Windsurf, Cline, Gemini, Antigravity). Its mechanisms transplant directly.
- Headroom is a network proxy/library (Rust core, Python bindings) that intercepts raw HTTP traffic to the LLM provider and compresses it (AST code compression, KV-cache alignment, cross-agent memory store, an MCP server). DevBureau ships markdown that the *host* IDE loads — it has no running process and no access to the request stream. Most of Headroom's actual mechanism cannot be replicated by a rules file; only the underlying *idea* (verbosity costs real money) transfers, not the technique.

### Findings

| Finding | Verdict | Reason |
|---|---|---|
| Ponytail's 7-rung "lazy ladder" (YAGNI → reuse existing code → stdlib → native platform feature → already-installed dependency → one-liner → minimum that works), run *after* understanding the problem, never *instead of* | **Adopt** | Directly reduces code-as-output tokens (their agentic benchmark: -54% LOC, -22% tokens, -20% cost, -27% time, 100% safety retained) without sacrificing quality — it explicitly protects trust-boundary validation, error handling, security, and accessibility, which matches DevBureau's own Code Quality Standards rather than conflicting with them. DevBureau's `clean-code` skill currently states *style* rules (naming, function size) but has no decision procedure for *whether to build something at all* or *reach for a dependency vs. stdlib* — this fills that gap. |
| Output discipline: code first, then ≤3 short lines (what was skipped, when to revisit), no essays explaining a simplification, full explanation only when explicitly asked | **Adopt** | This is the single highest-leverage lever available: output tokens cost ~5x input on Opus-class models (Headroom's own stated rationale), and DevBureau's GEMINI.md TIER 0 currently only enforces terse-output discipline inside the personal PT-BR "Gabarito" section, not as a universal, language-agnostic baseline for every agent regardless of language. Adding it at the universal level (not the personal layer) benefits every user of the published kit, not just this workspace. |
| `ponytail:` deliberate-shortcut comment convention (names the ceiling + upgrade trigger) plus a one-shot debt-ledger command that greps for the marker and reports anything with no named trigger | **Adopt** | DevBureau has no convention for marking "this is intentionally minimal, here's when to revisit" — shortcuts currently either don't get taken (over-engineering by default caution) or get taken silently (real debt with no trail). This closes that gap cheaply: a comment convention + one grep-based report skill, no new infrastructure. |
| Ponytail's `/ponytail-review` (diff-scope) and `/ponytail-audit` (repo-scope) over-engineering finder, tagged `delete:`/`stdlib:`/`native:`/`yagni:`/`shrink:`, explicitly out of scope for correctness/security (routes those to a normal review) | **Adopt** | Concrete, low-cost, clean separation of concerns from DevBureau's existing security/lint scripts — this is a complexity-only lens nothing in the current catalog provides. |
